# 메모리 시스템 500개 Fact 스케일 벤치마크 보고서

> 날짜: 2026-04-01
> 이슈: #189
> 벤치마크: 500개 fact, 106개 쿼리, 키워드 심판
> 테스트 LLM: Gemini 2.5 Flash, Qwen3-8B (ollama 로컬)

## 요약

Naia 메모리 시스템이 500개 fact 환경에서 **핵심 정확도 70-72%**를 기록했습니다. 100개 fact의 88%에서 하락했지만 예상된 결과입니다 — 500개에서 topK=10은 **2% 검색비율**로, 훨씬 높은 검색 정밀도가 필요합니다. Gemini와 Qwen3-8B가 거의 동일한 결과(±2%p)를 보여 벤치마크가 LLM에 의존하지 않음을 확인했습니다.

## 핵심 발견

### 1. LLM 독립성 확인
Gemini(70%)와 Qwen3(72%)가 ±2%p 이내 — 벤치마크는 **메모리 시스템 품질**을 측정하며 LLM 능력에 의존하지 않습니다. Qwen3-8B(무료 로컬)가 벤치마크용으로 충분합니다.

### 2. 스케일 영향 분석
100 → 500 fact에서 가장 큰 하락:
- **시맨틱 검색**: 86% → 57% (-29%p) — 벡터 결과에 노이즈 증가
- **직접 기억 인출**: 100% → 78-83% (-17-22%p) — 유사 fact 경쟁
- **개체 구분**: 88% → 63-75% (-13-25%p) — 더 많은 개체 혼동
- **환각 방지**: 92% → 67% (-25%p) — 기억이 많을수록 거짓 관련성 증가

### 3. 안정적 카테고리
- **무관 정보 격리**: 100% → 100% (모든 스케일에서 안정)
- **능동적 기억 적용**: 70% → 60-70% (최소 하락)

### 4. 500+ 스케일 개선 필요 영역
1. **벡터 검색 정밀도**: mem0가 스케일에서 너무 많은 느슨한 결과 반환
2. **Fact 중복 제거**: 유사 fact (예: "Neovim 에디터" vs "lazy.nvim 플러그인") 혼동
3. **시간 추론**: 0-25% — 사실상 미작동, 전용 시간 인덱스 필요

## 인프라

- **500개 fact**: 10개 도메인 (~50개씩), 44개 잡음 메시지, 15개 업데이트
- **Qwen3-8B**: ollama, Q4_K_M 양자화, ~5GB VRAM, ~53 tok/s
- **Gemini 2.5 Flash**: API, 요청당 2초 제한
- **총 벤치마크 시간**: LLM당 ~45분 (500 fact 인코딩 + 106 쿼리)

## 권장사항

1. **500 fact에서 80%+ 달성** 후 대규모 프로덕션 준비 판정
2. **벡터 검색 필터링 개선**: 관련성 threshold 또는 re-ranking 필요
3. **시간 인덱스 추가**: 현재 키워드/벡터 검색으로는 시간 기반 쿼리 불가
4. **runs=3 다수결**: 단일 실행은 LLM 비결정성으로 분산 큼
5. **하이브리드 검색 고려**: 키워드 사전 필터 + 벡터 재순위로 스케일 정밀도 확보
